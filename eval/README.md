# Evaluation assets

## `golden_set.jsonl` — the hand-verified golden set

40 questions over the **2025 Silverstone Grand Prix (Race)** corpus — the project's
most valuable asset and the ground truth every metric is computed against.

### Scope
The live database holds four races, but the eval is deliberately scoped to
**Silverstone only** (35 chunks: 15 stint + 20 result docs). Every eval run filters
`grand_prix=Silverstone` on both retrieval legs, so the other races cannot leak in.
See `docs/AUDIT.md §5`.

### Question distribution (designed for specific failure modes)
| Bucket | Count | Why it's here |
|---|---|---|
| `factual` | 8 | Baseline must pass these; a miss means something is broken upstream. |
| `comparative` | 10 | Two-driver synthesis. Naive dense top-k tends to return k chunks about **one** driver — the failure hybrid + reranking should fix. |
| `exact_term` | 8 | Driver codes (`VER`), compounds (`SOFT`/`HARD`), trap codes (`ST`). Dense embeddings are weak on exact tokens — **this is the gap BM25 exists to close.** |
| `paraphrase` | 8 | Vague natural language with little lexical overlap ("came out on top", "seven-time champion at his home race"). BM25 fails these; dense wins. Justifies **fusion**, not replacement. |
| `unanswerable` | 6 | Weather, qualifying, another year, pit-stop laps, setup. Tests groundedness — the system must refuse, not hallucinate. |

### Schema (one JSON object per line)
```json
{
  "id": "c_03",
  "question": "Between Norris and Piastri, who had the faster best lap ...?",
  "ground_truth": "Piastri (1:29.337) was faster than Norris (1:29.734).",
  "relevant_chunk_ids": ["2025_Silverstone_R_NOR_stint3", "2025_Silverstone_R_PIA_stint3"],
  "query_type": "comparative",
  "difficulty": "easy | medium | hard",
  "notes": "why it's interesting / the failure mode it probes"
}
```
`relevant_chunk_ids` reference the **deterministic** `chunk_id` in each document's
metadata (`{year}_{gp}_{session}_{driver}_stint{n}` or `..._result`), which is
stable across re-ingests — unlike the DB's autogenerated `id`. Unanswerable
questions carry an empty `relevant_chunk_ids`.

### How it was built (honest account)
1. **Corpus is deterministic.** Each chunk is generated from FastF1 lap/result data
   by `backend/ingestion.py` — no LLM in the corpus. So the chunk text *is* verified
   source data.
2. **Questions + answers were authored and verified by hand** against that corpus
   (finishing order, grid slots, stint compounds, lap times, trap speeds all
   cross-checked). `scripts/generate_candidates.py` is provided as an LLM *drafting*
   aid, but its output is explicitly `UNVERIFIED` and was **not** trusted blind —
   an LLM writing both questions and answers with no ground truth is the closed loop
   this eval is designed to avoid.
3. **Every label is machine-checked** by `eval/validate_golden.py`: schema, unique
   ids, valid buckets, and that every `relevant_chunk_id` actually exists in the
   live Silverstone corpus (no dangling labels after a re-ingest).

### Validate
```bash
python eval/validate_golden.py   # exits non-zero on any problem; CI gates on it
```
