# The Virtual Paddock Engineer — Code Walkthrough

A file-by-file explanation of how this F1 RAG application works, written so you
can understand and **own** every part of it.

Read it top to bottom: the order follows how data actually flows through the
system, so each file builds on the one before it.

---

## The 10,000-foot view

There are **two completely separate features** in this app:

1. **Chat (RAG)** — ask questions in English ("How is Hamilton so fast?") and get
   a grounded answer generated by a local LLM, with the source data shown as
   "receipts."
2. **Compare Telemetry (no RAG)** — pick two drivers and overlay their fastest-lap
   speed/throttle/brake/racing-line charts. This pulls raw numbers straight from
   FastF1 and never touches the LLM or the vector database.

```
                          ┌─────────────────────────────────────────┐
   React Frontend         │            FastAPI Backend               │
                          │                                          │
   Chat tab ────POST────► │  /api/chat/stream                        │
                          │     ├─ drivers.py  (detect "HAM")         │
                          │     ├─ vectorstore (search Supabase) ◄────┼──► Supabase
                          │     └─ chain.py    (LLM writes answer) ◄──┼──► Ollama
                          │                                          │
   Compare tab ──GET────► │  /api/telemetry                          │
                          │     └─ telemetry.py (raw FastF1 data) ◄───┼──► FastF1
                          └─────────────────────────────────────────┘
```

The chat feature is the RAG pipeline. The compare feature is a plain data API.
Keep that split in your head and the whole codebase makes sense.

---

## What "RAG" means here

RAG = **Retrieval-Augmented Generation**. The LLM (`llama3.2`) doesn't know
anything about the 2025 Silverstone GP. So instead of asking it to answer from
memory (it would hallucinate), we:

1. **Retrieve** the most relevant facts from our database, and
2. **Generate** an answer by handing those facts to the LLM as "context."

The LLM is only allowed to answer *using the facts we gave it*. That's why answers
are trustworthy — they're grounded in real stint data we ingested.

The two halves of RAG live in two phases:

- **Ingestion (offline, run once):** turn F1 data into searchable text + vectors.
  → `ingestion.py`
- **Query (online, every chat):** find relevant text, feed it to the LLM.
  → `chain.py` + `api.py`

## File 1: `ingestion.py` — loading the library

**Job:** read lap data from FastF1, turn each driver's stint into an English
sentence, embed it, and store it in Supabase. This is the only file that *writes*
to the database. You run it manually; it does not run during a chat.

### Why we turn data into sentences

A vector search matches by **meaning**. To search by meaning, the thing you store
must be *text*, because the embedding model turns text → vector. So we don't store
raw numbers — we write a natural-language **summary** of each stint, with the
numbers baked into the prose. That sentence is what gets embedded and searched.

### Walkthrough

**1. Cache setup (must be first):**

```python
CACHE_DIR = Path(__file__).resolve().parent / "f1_cache"
CACHE_DIR.mkdir(exist_ok=True)
fastf1.Cache.enable_cache(str(CACHE_DIR))
```

FastF1 downloads race data from the internet — slow and rate-limited. This points
it at a local folder so repeat runs read from disk. **It must run before any other
FastF1 call**, otherwise the cache is ignored.

**2. Formatting helpers (`fmt_laptime`, `fmt_sector`, `trap_speed`):**

Raw FastF1 values aren't human-readable. `LapTime` is a pandas `Timedelta`
(`0 days 00:01:28.412`), not `1:28.412`. These helpers make numbers pretty for the
sentence and guard against missing data (rendering `"n/a"` when a value is absent).

**3. The heart — `build_stint_documents()`:**

```python
session = fastf1.get_session(year, gp, session_type)
session.load(telemetry=False, weather=False)   # only lap data — fast
laps = session.laps.pick_quicklaps()            # drop in/out/safety-car laps
```

`pick_quicklaps()` matters: pit and safety-car laps are 2× normal pace; leaving
them in would pollute every average. We filter to clean racing laps *first*.

Then the **double loop** — this is our "chunking":

```python
for driver in laps["Driver"].unique():
    driver_laps = laps.pick_drivers(driver)
    for stint_num, stint_laps in driver_laps.groupby("Stint"):
        if len(stint_laps) < 3:
            continue                            # too short to be meaningful
```

A **stint** = the laps run on one set of tyres (between pit stops).
`groupby("Stint")` splits each driver's race into those chunks. **One stint → one
document.** We never split text by length — the data structure defines the chunk.

Inside, it computes the stats, including the **tyre-degradation slope**:

```python
x = fit["LapNumber"].to_numpy(dtype=float)
y = fit["LapTime"].dt.total_seconds().to_numpy(dtype=float)
degradation = float(np.polyfit(x, y, 1)[0])     # slope = seconds lost per lap
```

`np.polyfit(x, y, 1)` fits a straight line `y = mx + c`; `[0]` is the slope `m`.
Positive = tyres slowing over the stint; negative = pace improving.

Then it builds the sentence and the `Document` — the **single most important idea**
in the project:

```python
docs.append(Document(
    page_content=summary,    # ← the English sentence: EMBEDDED and searched by MEANING
    metadata={               # ← exact labels: used for EXACT filtering only
        "driver": driver, "year": year, "grand_prix": gp,
        "session_type": session_type, "stint": int(stint_num), "compound": compound,
    },
))
```

- `page_content` → becomes a vector → matched semantically ("worst degradation?").
- `metadata` → never embedded → used for exact `WHERE driver = 'HAM'` filtering.

**4. `build_result_documents()`** does the same pattern for race *results* (who
won, finishing positions, gaps). Note its metadata has `"doc_type": "result"` —
that flag is what the frontend uses to label a receipt "Result" instead of a stint.

**5. `main()`** runs only via `python -m backend.ingestion` (the
`if __name__ == "__main__"` guard). It does the actual embed + insert:

```python
SupabaseVectorStore.from_documents(
    docs, OllamaEmbeddings(model="nomic-embed-text"),
    client=supabase, table_name="documents", query_name="match_documents",
)
```

That one call embeds every sentence and inserts content + metadata + vector. The
guard exists so other files can `import build_stint_documents` without connecting
to the database.

**Mental model:**
`FastF1 race → loop (driver, stint) → write sentence + labels → embed + store`

## File 2: `vectorstore.py` — the shared database connection

**Job:** one small function that builds the connection to Supabase so every other
file can reuse it instead of repeating the wiring.

```python
def get_vector_store() -> SupabaseVectorStore:
    try:
        from dotenv import load_dotenv
        load_dotenv()                                  # pull SUPABASE_* from .env if present
    except ModuleNotFoundError:
        pass
    supabase = create_client(
        os.environ["SUPABASE_URL"],
        os.environ["SUPABASE_SERVICE_KEY"],            # service key — bypasses RLS
    )
    return SupabaseVectorStore(
        client=supabase,
        embedding=OllamaEmbeddings(model="nomic-embed-text"),
        table_name="documents", query_name="match_documents",
    )
```

Three pieces glued together:

- **`client`** — the Supabase connection (URL + key).
- **`embeddings`** — the *same* model used at ingestion. This is critical: you must
  search with the same embedding model you stored with, or the vectors live in
  different "spaces" and nothing matches.
- **`SupabaseVectorStore`** — LangChain's wrapper that knows how to call the
  `match_documents` SQL function to do the similarity search.

`query_name="match_documents"` points at the SQL function defined in the database
(see CLAUDE.md Phase 1a). The table alone lets you *insert* but not *search* — the
function is what runs the `1 - (embedding <=> query)` cosine-similarity query.

This file is reused by both `chain.py` and `api.py`. It exists purely to avoid
duplicating connection code.

## File 3: `chain.py` — the RAG brain (retrieve + ask the LLM)

**Job:** the standalone RAG pipeline. Given a question, retrieve relevant stints
and have the LLM write a grounded answer. This is the logic the API later wraps.

### The LCEL chain

LangChain Expression Language (LCEL) composes steps with `|`, like a Unix pipe:

```python
def build_chain():
    retriever = get_vector_store().as_retriever(search_kwargs={"k": RETRIEVAL_K})
    llm = ChatOllama(model="llama3.2", temperature=0)
    return (
        {"context": retriever | format_docs, "question": RunnablePassthrough()}
        | PROMPT | llm | StrOutputParser()
    )
```

(`RETRIEVAL_K = 4` — we fetch the top 4 matching stints.) `format_docs` joins the
retrieved documents' sentences into one block of text:

```python
def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)
```

Read it as a pipeline. When you call `chain.invoke("worst degradation?")`:

1. The **front dict** runs two branches in parallel to fill the prompt variables:
   - `"context"`: `retriever` finds the top-`k` matching stint documents, then
     `format_docs` joins their sentences into one block of text.
   - `"question"`: `RunnablePassthrough()` just forwards the original question
     unchanged.
2. The dict resolves to `{"context": "...", "question": "..."}`.
3. `PROMPT` slots those into the template.
4. `llm` (`ChatOllama`) generates the answer.
5. `StrOutputParser()` extracts the plain string from the LLM's response object.

### The prompt — strict grounding

The real `GROUNDING_INSTRUCTION` (shared with the API) is longer and does two jobs
at once — it forbids outside knowledge **but explicitly allows reasoning over the
numbers it was given**:

```python
GROUNDING_INSTRUCTION = (
    "You are a concise Formula 1 race engineer. Answer using ONLY the stint data in the "
    "context below. You MAY compare, rank, and reason over those numbers — lap times, "
    "sector times, speed-trap figures, tyre degradation and compounds — to explain things "
    "like who was faster and why ... Do not bring in any knowledge beyond the context; if "
    "the context lacks what's needed to answer, say so plainly (e.g. \"That isn't in the "
    "data I have.\"). Ground every claim in specific numbers from the context."
)
```

This is what makes the bot trustworthy: it's *forbidden* from answering out of its
own training memory (ask "who won the 1998 Monaco GP?" and it refuses), yet it's
*encouraged* to compare and rank the retrieved stints to answer "who was faster and
why." `temperature=0` makes it deterministic (same question → same answer), which
you want for a factual tool.

> Note the limit on "reasoning": the LLM can only reason as well as `llama3.2` (a
> small local model) allows, and only over the 4 stints retrieved. Improving
> answers means better retrieval (multi-query, hybrid search, reranking) **and/or**
> a stronger model — the prompt already permits the reasoning.

### One gotcha to remember

`StrOutputParser()` throws away the retrieved documents — it keeps only the text
answer. But the UI needs those documents to show "receipts." So in `api.py` we
**retrieve separately** rather than relying on this chain to surface the docs.

`chain.py` is runnable on its own (`python -m backend.chain "your question"`),
which is how Phase 2 was tested before any web server existed.

## File 4: `drivers.py` — turning "Hamilton" into "HAM"

**Job:** read a user's free-text question and figure out which driver they mean, so
we can filter the search to just that driver. The stored metadata uses 3-letter
codes (`HAM`, `VER`), but users type "Hamilton", "lewis", or "lando".

### How the alias map is built

```python
for _, row in session.results.iterrows():
    code = str(row["Abbreviation"]).upper()
    for raw in (row["FullName"], row["LastName"], row["FirstName"]):
        aliases[key] = code     # {"hamilton": "HAM", "lewis": "HAM", ...}
```

It pulls every driver's full name / last name / first name from FastF1 results and
maps each (lowercased) to their code. The result is cached to
`driver_aliases.json` so the API never has to load FastF1 at startup — it just
reads the JSON. **You regenerate this cache after ingesting new races**
(`python -m backend.drivers`).

### How detection works (`detect_drivers`)

Two passes:

```python
for token in re.findall(r"\b[A-Z]{3}\b", text):    # explicit codes: "HAM", "VER"
    if token in codes: add(token)

lowered = text.lower()
for alias in sorted(alias_map, key=len, reverse=True):   # longest aliases first
    if re.search(rf"\b{re.escape(alias)}\b", lowered): add(alias_map[alias])
```

- First it catches explicit uppercase codes in the raw text.
- Then it matches name aliases, **longest first** so "lewis hamilton" counts as one
  match (HAM), not two.

### The two functions and why both exist

- **`detect_drivers`** (plural) → returns *every* driver mentioned. Used for
  comparisons.
- **`detect_driver`** (singular) → returns one code only if *exactly one* driver is
  mentioned, else `None`. `None` means "don't filter — search across everyone,"
  which is correct for broad questions ("who had the worst tyres?") and for
  two-driver comparisons.

### A subtle design choice

`"max"` is in the stopword list. Why? Our stint sentences say "max speed", which
would falsely match Verstappen. So "max" is ignored as an alias — but "verstappen"
and "VER" still detect him fine.

## File 5: `telemetry.py` — raw data for the charts (NO RAG)

**Job:** for the Compare tab, fetch a driver's fastest-lap telemetry (speed,
throttle, brake, track position) straight from FastF1 and return it as JSON. This
**bypasses the LLM and the vector database entirely** — it's a plain data API.

### Why no RAG here?

The chart needs *exact* numbers, not a generated approximation. "Give me VER's
fastest lap speed trace" has one correct answer and requires no language. RAG would
add cost and fuzziness for zero benefit.

### Walkthrough

**Caching at two levels** — telemetry loads are expensive, so:

```python
_SESSION_CACHE = {}    # loaded FastF1 sessions, keyed by (year, gp, session)
_SCHEDULE_CACHE = {}   # season calendars
```

These are in-memory dicts so repeated requests in the same server run are instant.

**`get_schedule()`** returns the full season calendar (round, name, location,
country) — this feeds the race dropdowns in both tabs.

**`fastest_lap_telemetry()`** is the core:

```python
lap = driver_laps.pick_fastest()       # the driver's single best lap
tel = lap.get_telemetry()              # speed/throttle/brake/X/Y vs Distance
grid = np.linspace(dist.min(), dist.max(), GRID_POINTS)   # 400 even points

def resample(col):
    return np.interp(grid, dist, tel[col])
```

The **resampling** is the clever bit. Two drivers record telemetry at different raw
sample points, so you can't overlay them directly. `np.interp` re-samples every
channel onto a **shared 400-point distance grid**, so at "metre 1,200" both drivers
have a comparable value. That's what lets the frontend plot them on one x-axis.

**`compare_telemetry()`** just loops over the requested drivers, calls
`fastest_lap_telemetry` for each, and collects results (plus any per-driver errors,
so one missing driver doesn't break the whole response).

## File 6: `api.py` — the web server that ties it together

**Job:** the FastAPI app. It exposes HTTP endpoints, wires together
`drivers.py` + `vectorstore` + `chain` logic + `telemetry.py`, and handles CORS and
streaming. This is the single entry point the frontend talks to.

### Startup (runs once)

```python
VECTOR_STORE = get_vector_store()
LLM = ChatOllama(model="llama3.2", temperature=0)
ALIAS_MAP = load_alias_map()
ANSWER_CHAIN = CHAT_PROMPT | LLM | StrOutputParser()
```

Everything expensive is built **once** at startup and reused on every request —
you don't want to reconnect to Supabase or reload the alias map per message.

Note `ANSWER_CHAIN` is `prompt | llm | parser` — it does **not** include the
retriever. That's deliberate (the gotcha from `chain.py`): we retrieve separately
so we can also return the documents as receipts.

### CORS

```python
app.add_middleware(CORSMiddleware,
    allow_origin_regex=r"http://(localhost|127\.0\.0\.1):\d+", ...)
```

Browsers block a page on `localhost:5173` from calling `localhost:8000` unless the
server explicitly allows it. The regex allows any localhost port (Vite hops ports
in dev). Tighten this for production.

### The request models

```python
class ChatRequest(BaseModel):
    message: str
    driver: str | None = None        # explicit override (else auto-detected)
    grand_prix: str | None = None    # dropdown filter
    session_type: str | None = None
    history: list[Turn] = []         # recent turns for follow-up memory
```

Pydantic validates the incoming JSON automatically — wrong shape → 422 error before
your code runs.

### The chat flow (the important endpoints)

Both `/api/chat` and `/api/chat/stream` follow the same three steps:

```python
driver = resolve_driver(req)                       # 1. who are they asking about?
docs   = retrieve(req.message, build_filter(...))  # 2. find relevant stints
answer = ANSWER_CHAIN.invoke({...})                # 3. LLM writes grounded answer
```

**Step 1 — `resolve_driver()`** decides the driver filter by priority:
1. explicit dropdown choice, else
2. detected from the current message (`drivers.py`), else
3. the most recent driver mentioned earlier in the conversation — so "what about
   *his* second stint?" keeps filtering on the right person.

**Step 2 — `build_filter()` + `retrieve()`** builds a JSONB-containment filter
(`{"driver": "HAM", "grand_prix": "Silverstone"}`) and runs the vector search with
it. Only keys that are set get included, so an unfiltered question searches
everyone.

**Step 3 — the LLM** answers from the retrieved context + conversation history.

### Streaming (`/api/chat/stream`) — SSE

The streaming endpoint sends Server-Sent Events so the UI updates live:

```python
async def event_stream():
    yield _sse({"type": "sources", "sources": docs_to_sources(docs)})  # receipts first
    async for tok in ANSWER_CHAIN.astream(payload):
        yield _sse({"type": "token", "text": tok})                     # then each word
    yield _sse({"type": "done"})                                       # then finish
```

It sends the **receipts first** (so the source panel fills immediately), then
streams the answer token-by-token, then a `done` marker. Each message is formatted
as `data: <json>\n\n` — the SSE wire format.

### The other endpoints

- **`/api/health`** — liveness check.
- **`/api/filters`** — distinct drivers / races / sessions that actually exist in
  the database, for the dropdowns and the "no ingested data" warning.
- **`/api/schedule`** — the full season calendar (from `telemetry.get_schedule`).
- **`/api/telemetry`** — calls `compare_telemetry` for the Compare tab.

## File 7: `api.js` — the thin client

**Job:** every network call to the backend lives here, so components never write
`fetch` themselves.

- **`fetchFilters()`**, **`fetchSchedule()`**, **`fetchTelemetry()`** — simple
  `GET` requests that return JSON.
- **`streamChat()`** — the interesting one. It consumes the SSE stream:

```python
const reader = res.body.getReader()
const decoder = new TextDecoder()
let buffer = ''
while (true) {
    const { value, done } = await reader.read()
    if (done) break
    buffer += decoder.decode(value, { stream: true })
    const events = buffer.split('\n\n')
    buffer = events.pop() ?? ''      // carry the trailing partial event over
    for (const evt of events) {
        const payload = JSON.parse(evt.slice(5).trim())   // strip "data:"
        if (payload.type === 'sources') onSources?.(payload)
        else if (payload.type === 'token') onToken?.(payload.text)
        else if (payload.type === 'done') onDone?.()
    }
}
```

Two things to understand:
- **Why not `EventSource`?** The browser's built-in SSE client is GET-only, but our
  endpoint is a POST (it has a request body). So we read the stream manually with
  `fetch` + a reader.
- **The buffering** is essential. Network chunks don't align to SSE event
  boundaries — a chunk might end mid-event. So we split on `\n\n`, process complete
  events, and **carry the leftover partial** (`buffer = events.pop()`) into the next
  read. The callbacks (`onSources`, `onToken`, `onDone`) push the data into the
  React component.

## File 9: `Chat.jsx` — the RAG tab

**Job:** the chat UI. Manage messages, send questions, stream answers, show
receipts.

### State

```python
const [messages, setMessages] = useState([])  // the conversation
const [answer, setAnswer]     = useState('')  // the in-progress streaming answer
const [sources, setSources]   = useState([])  // receipts for the current answer
const [driver, setDriver]     = useState(null)// which driver got auto-detected
const [streaming, setStreaming] = useState(false)
```

### Sending a message (`send`)

```python
const history = messages.slice(-6).map((m) => ({ role: m.role, content: m.text }))
// ... add the user's message, clear inputs ...
await streamChat(
    { message: q, grand_prix: gp || null, session_type: 'R', history },
    {
        onSources: (p) => { setSources(p.sources); setDriver(p.driver) },
        onToken:   (t) => { acc += t; setAnswer(acc) },
    }
)
setMessages((m) => [...m, { role: 'assistant', text: acc, driver: drv, sources: srcs }])
```

It sends the last 6 turns as `history` (lightweight memory for follow-ups), then
streams: `onSources` fills the receipts panel, `onToken` appends each word to the
live answer bubble. When the stream ends, the finished answer is saved into the
message list.

### The "no ingested data" warning

```python
const isIngested = ingested.includes(gp)
...
{!isIngested && gp && (
    <span className="warn">No ingested data for this race — chat will say so.</span>
)}
```

If you pick a race that isn't in the database, it warns you up front.

### The receipts panel

The `<aside className="receipts">` maps over `sources` and renders each retrieved
document's content + metadata. It checks `doc_type === 'result'` to label a receipt
"Result" vs showing "Stint N" + compound. This is the "show your work" panel that
proves the answer is grounded.

## File 10: `Compare.jsx` — the telemetry tab (NO RAG)

**Job:** pick a race + two drivers, fetch their fastest-lap telemetry, and draw
charts. Pure data visualization — no LLM involved.

### Fetching

```python
async function run() {
    const picks = a === b ? [a] : [a, b]
    const res = await fetchTelemetry({ drivers: picks, grand_prix: race, session_type: 'R' })
    setData(res)
}
```

### The charts (Recharts)

- **`mergeChannel()`** — zips the two drivers' arrays into rows
  `{ distance, VER: x, HAM: y }` aligned by sample index (they share the 400-point
  grid from `telemetry.py`).
- **`Trace`** — a reusable line chart, used for Speed and Throttle.
- **`SpeedDelta`** — computes `A.speed - B.speed` at each point; above zero means A
  is faster there, below means B. The single most useful comparison chart.
- **`TrackMap`** — draws the racing line from the X/Y position channels as an SVG
  path, scaling the coordinates to fit the box and inverting Y (screen Y grows
  downward, track Y grows upward).

# Part 3 — Putting it all together

## A chat request, end to end

```
1. You type "How is Hamilton so fast?" and hit Send          (Chat.jsx)
2. streamChat POSTs to /api/chat/stream                      (api.js)
3. resolve_driver: "hamilton" → HAM                          (api.py → drivers.py)
4. build_filter: { driver: "HAM", grand_prix: "Silverstone" }(api.py)
5. retrieve: vector search in Supabase with that filter      (vectorstore.py)
6. SSE "sources" event → receipts panel fills                (api.py → api.js → Chat.jsx)
7. LLM streams the answer token by token                     (chain logic in api.py → Ollama)
8. SSE "token" events → answer bubble grows word by word     (api.js → Chat.jsx)
9. SSE "done" → answer saved to message history              (Chat.jsx)
```

## A compare request, end to end

```
1. You pick Silverstone + VER + HAM, click Compare           (Compare.jsx)
2. fetchTelemetry GETs /api/telemetry?drivers=VER,HAM&...     (api.js)
3. compare_telemetry loads the session from FastF1 (cached)  (api.py → telemetry.py)
4. each driver's fastest lap is resampled to 400 points      (telemetry.py)
5. JSON returned                                             (api.py)
6. mergeChannel zips the arrays; Recharts draws 4 charts     (Compare.jsx)
```

## The one distinction to never forget

| | Chat tab | Compare tab |
|---|---|---|
| Uses the LLM? | **Yes** | No |
| Uses the vector DB? | **Yes** | No |
| Data source | Supabase (pre-ingested) | FastF1 (live) |
| Output | Generated English answer | Exact numbers / charts |
| Files | ingestion, vectorstore, chain, drivers, api | telemetry, api |

# Glossary

- **Embedding** — a list of 768 numbers representing the *meaning* of a piece of
  text. Similar meanings → similar vectors. Produced by `nomic-embed-text`.
- **Vector search** — finding the stored texts whose embeddings are closest (by
  cosine similarity) to the question's embedding.
- **pgvector** — the Postgres extension (in Supabase) that stores vectors and does
  the similarity search via the `match_documents` SQL function.
- **Document** — a LangChain object with `page_content` (embedded text) and
  `metadata` (exact-match labels).
- **Stint** — the laps a driver runs on one set of tyres, between pit stops. Our
  unit of "chunking."
- **Degradation slope** — seconds of lap time lost per lap, from a linear fit;
  positive = tyres fading.
- **LCEL** — LangChain Expression Language; composes steps with `|` like a pipe.
- **Grounding** — forcing the LLM to answer only from provided context, not its own
  training memory. Prevents hallucination.
- **SSE** — Server-Sent Events; a one-way stream of `data: ...\n\n` messages used
  to push answer tokens to the browser live.
- **CORS** — the browser security rule the backend must satisfy to let the React
  origin call it.
